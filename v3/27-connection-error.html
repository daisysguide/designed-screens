Decision: two patterns, not one screen
Variant A — Full screen: App launched with zero connectivity. Nothing can load. Show a centered offline state with retry CTA. No nav, no shell — the app hasn't loaded.
Variant B — Inline banner: User is mid-session when connection drops. The UI they're on stays visible in read-only mode. A non-blocking banner at the top informs them. Answers submitted while offline queue and sync on reconnect.
Don't force a full offline screen for mid-session drops — it's jarring and loses context. The inline banner pattern is less disruptive and honest about capability.

Variant B — Inline offline banner
TriggerNetwork connection lost while app is open and in use
Banner bgrgba(26,26,26,0.06) — neutral, low-contrast. Not error red — this isn't the user's fault and isn't blocking their current screen.
Banner copy"You're offline." bold + "Read-only for now — your answers will sync when you reconnect." · 13px · text-secondary
Banner positionTop of scroll area, above content — persistent until reconnected
CTA behaviorAny action requiring network (submitting an answer, syncing data) shows a toast: "Can't do that offline. Will sync when you're back." Answer CTAs are disabled / greyed out.
Read-only contentTopic articles, question text, and previously loaded content remain visible. Skeletons show for content that hasn't loaded yet.
Auto-dismissBanner dismisses and normal interaction resumes automatically when connection restores. No toast needed — just normal UX returns.
What this does NOT cover
Server errors (5xx) — handled by inline error state on the specific failed action, not this pattern. Copy: "Something went wrong on our end. Try again."
Slow connections — no special treatment in V1. Natural timeout handling by the network layer.
Session expiry / auth errors — handled by redirect to Log In (Screen 04), not this pattern.
Components used
Empty State Toast / Snackbar Bottom Nav Screen Header
App Store compliance note: App Store Review expects graceful offline handling — crashing or showing a blank screen on connectivity loss will fail review. Both variants satisfy this requirement. Variant B additionally satisfies the expectation that existing in-progress data isn't lost.
