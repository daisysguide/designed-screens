Behavior notes
One user interaction on this screen: tapping "Claim my plan →" on the done state. No back button, no bottom nav.
Steps complete in order: Step 1 → Step 2 → Step 3. Each active for ~700ms. Total animation: ~2–3 seconds.
When all 3 steps complete, the heading/body transitions to the done state, the spinner is replaced by the Daisy logo, the step list is hidden, and the "Claim my plan →" CTA appears.
API timing
Under 2 seconds: hold for the animation; don't skip ahead.
2 to 10 seconds: loop step 3 spinner until the API resolves, then transition to done state.
10 to 15 seconds: keep spinner looping and add a subtle line under step 3: "Taking a little longer than usual..." (DM Sans 13px, text-secondary).
Past 15 seconds, or on API error: route to Screen 28 (Connection Error). Retry path returns to Screen 03 and re-runs the sequence.
Accessibility
Step list container uses aria-live="polite" and aria-atomic="false" so each step transition is announced.
Heading/body region uses aria-live="polite" so the transition to the done state is announced.
Spinner has aria-hidden="true". The step copy carries the meaning.
Data persistence
An anonymous Supabase user record is created before this screen (likely on Screen 02 entry). Quiz answers are saved against the anonymous ID. On Screen 04 sign-up, the anonymous record is merged with the new permanent account.
Users who abandon between Screen 03 and Screen 04 can resume from Screen 08 on return.
Implementation flag for Jake: this affects the auth flow across Screens 02, 03, 04 and how returning unauthenticated users are routed.
